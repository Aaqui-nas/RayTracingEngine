# TP29 — Post-processing : tone mapping, bloom et SSAO

**Durée estimée : ~8h**
**C++ introduit :** `std::span`, pipeline fonctionnel avec lambdas, convolution de kernels
**Feature graphique :** Tone mapping ACES/Reinhard, bloom (éclat lumineux), occlusion ambiante screen-space

---

## Objectif

Le ray tracer génère une image en **HDR** (High Dynamic Range) — les valeurs de luminance peuvent dépasser 1.0. L'écran, lui, ne peut afficher que des valeurs entre 0 et 1 en LDR (Low Dynamic Range).

Le **post-processing** est l'ensemble des passes appliquées sur le buffer HDR **avant** la conversion finale en pixels :

1. **Tone mapping** — comprimer le HDR en LDR en préservant la perception
2. **Bloom** — les zones très lumineuses "débordent" visuellement sur les voisins
3. **SSAO** — Screen-Space Ambient Occlusion : assombrit les coins et recoins

Ces passes s'enchaînent comme un pipeline fonctionnel.

---

## 1. std::span

`std::span<T>` est une vue non-propriétaire sur un tableau contigu. Il ne possède pas les données.

```cpp
#include <span>

void process(std::span<float> pixels) {
    for (float& p : pixels) p *= 2.0f;
}

std::vector<float> v = {1, 2, 3, 4};
process(v);                    // OK — vector est contigu
process({v.data(), 2});        // seulement les 2 premiers
```

Avantage vs `vector<T>&` : ne copie rien, ne transfère pas la propriété, peut pointer sur la moitié d'un buffer.

Avantage vs `T* + size` : `span` encapsule le pointeur et la taille ensemble — pas de risque d'oublier la taille.

## 2. Représentation du buffer HDR

struct PixelHDR {

    int width, height;
    std::vector<PixelHDR> pixels;

    PixelHDR& at(int x, int y) { return pixels[y * width + x]; }
    const PixelHDR& at(int x, int y) const { return pixels[y * width + x]; }
    std::span<PixelHDR> row(int y) { return {pixels.data() + y * width, (size_t)width}; }

## 3. Tone mapping

### Reinhard (1 paramètre)

Compression douce qui comprime [0, +∞) → [0, 1) :

```
L_out = L_in / (1 + L_in)
```

Avec un paramètre de luminosité blanche `L_white` :
```
L_out = L_in × (1 + L_in / L_white²) / (1 + L_in)
```

**Propriétés :** préserve les relations tonales, un peu terne pour les couleurs vives.

### ACES (Academy Color Encoding System)

L'opérateur utilisé dans Unreal Engine, Cyberpunk, beaucoup de jeux AAA :

```
a = 2.51, b = 0.03, c = 2.43, d = 0.59, e = 0.14
ACES(x) = clamp((x × (a×x + b)) / (x × (c×x + d) + e), 0, 1)
```

**Propriétés :** contraste plus marqué, teintes plus vibrantes, point blanc plus propre.

### Gamma correction

Après le tone mapping, applique la correction gamma (les écrans ont un espace perceptuel non-linéaire) :

```
L_gamma = pow(L_linear, 1.0 / 2.2)
```

## Exercice 1 — Pipeline de post-processing (45 min)

Crée `src/core/post_process.h`. L'idée : une liste de passes s'appliquant séquentiellement sur le buffer.

```cpp
using PostPass = std::function<void(ImageBuffer&)>;

class PostPipeline {
    std::vector<PostPass> passes;

public:
    PostPipeline& add(PostPass pass) {
        passes.push_back(std::move(pass));
        return *this;   // chaînable : pipeline.add(tone_map).add(gamma)
    }

    void apply(ImageBuffer& buf) const {
        for (const auto& pass : passes)
            pass(buf);
    }
};
```

**Usage prévu :**
```cpp
PostPipeline pipeline;
pipeline
    .add(bloom_pass(threshold=1.5, radius=5, strength=0.3))
    .add(ssao_pass(radius=0.5, samples=16))
    .add(tone_map_aces())
    .add(gamma_correct(2.2));

pipeline.apply(hdr_buffer);
```

## Exercice 3 — Convolution générique (60 min)

Bloom et SSAO utilisent des **filtres de convolution** : chaque pixel est remplacé par une somme pondérée de ses voisins.

```
pixel_sortie[x][y] = Σ kernel[dx][dy] × pixel_entree[x+dx][y+dy]
                     (dx,dy) ∈ [-r, r]²
```

Implémente une fonction générique :

```cpp
ImageBuffer convolve(
    const ImageBuffer& src,
    std::span<const double> kernel,   // kernel[ky*(2r+1)+kx]
    int radius);                      // kernel est (2r+1)×(2r+1)
```

**Gestion des bords :** utilise le pixel le plus proche (clamp) ou ignore les pixels hors bords (ne les compte pas dans la somme).

**Kernel gaussien :**
```
G(x, y, sigma) = exp(-(x²+y²) / (2σ²)) / (2π σ²)
```

## Exercice 4 — Bloom (60 min)

Le bloom simule la diffusion de la lumière dans le système optique de l'œil ou de l'appareil photo.

**Algorithme simplifié :**

```
1. Extraire les pixels "brillants" : bright_mask[x][y] = max(0, pixel - threshold)
2. Flouter le masque avec un kernel gaussien (radius 5–15)
3. Additionner au buffer original avec un coefficient strength
```

```cpp
PostPass bloom_pass(double threshold = 1.5, int radius = 7, double strength = 0.3) {
    return [threshold, radius, strength](ImageBuffer& buf) {
        // 1. Crée bright_buffer : pixel si luminance > threshold, sinon 0
        ImageBuffer bright = extract_bright(buf, threshold);

        // 2. Blur gaussien (séparable : horizontal puis vertical)
        ImageBuffer blurred = blur_horizontal(bright, radius);
        blurred = blur_vertical(blurred, radius);

        // 3. Additionne
        for (int y = 0; y < buf.height; y++)
            for (int x = 0; x < buf.width; x++)
                buf.at(x, y) += blurred.at(x, y) * strength;
    };
}
```

**Astuce d'optimisation :** le kernel gaussien est **séparable** : flouter horizontalement puis verticalement donne le même résultat qu'un flou 2D, mais en O(r × N) au lieu de O(r² × N). C'est crucial pour un grand rayon.

## Exercice 5 — SSAO (Screen-Space Ambient Occlusion) (90 min)

SSAO assombrit les zones où la géométrie est "coincée" — coins, recoins, jonctions. Il simule la lumière ambiante qui ne peut pas atteindre certains endroits.

**Principe :**  
Pour chaque pixel, on sait sa position 3D et sa normale (depuis le G-buffer ou depuis le ray tracer). On tire `N` échantillons aléatoires dans l'hémisphère autour de la normale, et on teste si ces points sont "occlus" par la géométrie voisine.

**Dans le contexte du ray tracer :**

Le ray tracer ne génère pas de G-buffer nativement. Deux approches :

**Approche A (simple)** : approximation screen-space basée sur la profondeur.  
On enregistre le buffer de profondeur `depth[x][y] = t_hit` lors du rendu. Le SSAO estime l'occlusion depuis les différences de profondeur locales.

**Approche B (correcte)** : enregistre aussi `normal[x][y]` lors du rendu. Pour chaque pixel, génère des directions aléatoires dans l'hémisphère de la normale, et cherche dans le buffer si d'autres pixels sont à portée.

**Algorithme SSAO simplifié (approche A) :**

```
Pour chaque pixel (x, y) :
    depth = depth_buffer[x][y]
    occlusion = 0

    Pour k = 1..N_SAMPLES :
        (dx, dy) = direction aléatoire dans disque de rayon max_radius
        depth_voisin = depth_buffer[x+dx][y+dy]

        Si depth_voisin < depth (le voisin est plus proche de la caméra = devant)
        et depth - depth_voisin < depth_threshold (pas trop loin = même objet) :
            occlusion += 1

    ao = 1 - occlusion / N_SAMPLES
    buffer[x][y] *= ao   // assombrit le pixel
```

**Modification du renderer pour capturer la profondeur :**

```cpp
struct RenderOutput {
    ImageBuffer color;
    std::vector<double> depth;   // depth[y * width + x] = t_hit (ou +inf si miss)
    std::vector<Vec3d> normals;  // pour SSAO correct
};
```
