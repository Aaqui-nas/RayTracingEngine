# TP28 — Thread pool avancé et rendu en tuiles

**Durée estimée : ~8h**
**C++ introduit :** `std::future`, `std::promise`, `std::condition_variable`, `std::packaged_task`
**Feature graphique :** Tile-based rendering (tuiles 16×16), load balancing adaptatif

---

## Objectif

Le renderer actuel découpe le travail **par lignes** : chaque thread prend une ligne de pixels. C'est simple, mais inefficace :
- Les lignes du haut (ciel, fond vide) se calculent en ~1 ms, les lignes du bas (sols chargés) en ~50 ms
- Les derniers threads finissent longtemps après les premiers — le CPU est sous-utilisé
- Aucune priorité possible

Ce TP construit un vrai **thread pool** avec une file de tâches partagée. Le rendu est découpé en **tuiles** de 16×16 pixels. Chaque thread prend la prochaine tuile disponible dès qu'il est libre.

## 1. Primitives de synchronisation C++

### `std::mutex` — exclusion mutuelle (déjà vu en TP10)

```cpp
std::mutex m;
{
    std::lock_guard<std::mutex> lock(m);
    // section critique
}   // lock libéré ici automatiquement (RAII)
```

### `std::condition_variable` — attente sur une condition

Permet à un thread d'attendre **efficacement** qu'une condition soit vraie, sans busy-wait :

```cpp
std::mutex m;
std::condition_variable cv;
bool ready = false;

// Thread producteur
{
    std::lock_guard<std::mutex> lock(m);
    ready = true;
    cv.notify_one();   // réveille un thread en attente
}

// Thread consommateur
{
    std::unique_lock<std::mutex> lock(m);
    cv.wait(lock, []{ return ready; });  // attend jusqu'à ce que ready == true
    // ici, ready est vrai et le mutex est re-acquis
}
```

`cv.wait(lock, pred)` : libère le mutex, attend, puis le re-acquiert quand on est réveillé et que `pred()` est vrai. Sans `pred`, il y a le risque de **spurious wakeup** (réveil sans raison) — la condition peut être fausse même après `notify`.

### `std::future` et `std::promise` — communication entre threads

```cpp
std::promise<int> p;
std::future<int> f = p.get_future();

std::thread t([&p]() {
    int result = 42;
    p.set_value(result);   // "envoie" la valeur
});

int val = f.get();   // bloque jusqu'à ce que la valeur soit disponible
t.join();
```

`std::promise` est le côté écriture, `std::future` le côté lecture.

### `std::packaged_task` — emballer une fonction pour en obtenir un future

```cpp
std::packaged_task<int(int, int)> task([](int a, int b) { return a + b; });
std::future<int> f = task.get_future();

std::thread t(std::move(task), 3, 4);
int result = f.get();   // = 7
t.join();
```

## 2. Architecture d'un thread pool

```
ThreadPool
├── vector<thread>           ← threads workers
├── queue<function<void()>>  ← file de tâches
├── mutex                    ← protège la file
├── condition_variable        ← signale l'ajout de tâches
└── bool stopped              ← signal d'arrêt

submit(task) :
    ajoute task dans la queue
    notify_one() pour réveiller un worker

Worker loop (chaque thread) :
    while (!stopped) :
        lock(mutex)
        wait(cv, "queue non vide ou stopped")
        prend la première tâche
        unlock
        exécute la tâche
```

## Exercice 1 — Thread pool minimal (60 min)

Crée `src/core/thread_pool.h` :

```cpp
class ThreadPool {
    std::vector<std::thread> workers;
    std::queue<std::function<void()>> tasks;
    std::mutex queue_mutex;
    std::condition_variable cv;
    bool stopped = false;

public:
    explicit ThreadPool(std::size_t num_threads);
    ~ThreadPool();   // attend que toutes les tâches soient terminées

    // Soumet une tâche et retourne un future pour son résultat
    template<typename F, typename... Args>
    auto submit(F&& f, Args&&... args) -> std::future<std::invoke_result_t<F, Args...>>;

    // Attend que toutes les tâches en cours soient terminées
    void wait_all();
};
```

**Corps de chaque worker :**

```cpp
// Chaque thread exécute cette boucle
while (true) {
    std::function<void()> task;
    {
        std::unique_lock<std::mutex> lock(queue_mutex);
        cv.wait(lock, [this]{ return stopped || !tasks.empty(); });
        if (stopped && tasks.empty()) return;
        task = std::move(tasks.front());
        tasks.pop();
    }
    task();   // exécute hors du lock
}
```

**`submit` avec `packaged_task` :**

```cpp
template<typename F, typename... Args>
auto submit(F&& f, Args&&... args) -> std::future<...> {
    using RetType = std::invoke_result_t<F, Args...>;
    auto task = std::make_shared<std::packaged_task<RetType()>>(
        std::bind(std::forward<F>(f), std::forward<Args>(args)...));
    auto fut = task->get_future();
    {
        std::lock_guard<std::mutex> lock(queue_mutex);
        tasks.push([task]() { (*task)(); });
    }
    cv.notify_one();
    return fut;
}
```

**Test :**
```cpp
ThreadPool pool(4);
auto f1 = pool.submit([](int x) { return x * x; }, 5);
auto f2 = pool.submit([](int x) { return x + 1; }, 10);
assert(f1.get() == 25);
assert(f2.get() == 11);
```

## Exercice 2 — Tile-based renderer (90 min)

Au lieu d'une ligne par tâche, utilise des tuiles de `tile_w × tile_h` pixels.

**Structure d'une tuile :**
```cpp
struct Tile {
    int x0, y0;   // coin supérieur gauche
    int x1, y1;   // coin inférieur droit (exclu)
};
```

**Génération des tuiles :**
```cpp
std::vector<Tile> make_tiles(int width, int height, int tile_w, int tile_h) {
    std::vector<Tile> tiles;
    for (int y = 0; y < height; y += tile_h)
        for (int x = 0; x < width; x += tile_w)
            tiles.push_back({x, y,
                             std::min(x + tile_w, width),
                             std::min(y + tile_h, height)});
    return tiles;
}
```

**Rendu en tuiles :**
```cpp
void render_tiles(const Scene& scene, const Camera& camera,
                  PixelBuffer& buffer, int samples, int tile_size)
{
    ThreadPool pool(std::thread::hardware_concurrency());
    auto tiles = make_tiles(buffer.width, buffer.height, tile_size, tile_size);

    std::vector<std::future<void>> futures;
    for (const Tile& tile : tiles) {
        futures.push_back(pool.submit([&scene, &camera, &buffer, &tile, samples]() {
            render_tile(scene, camera, buffer, tile, samples);
        }));
    }

    for (auto& f : futures) f.get();   // attend la fin
}
```

## Exercice 4 — Ordre de rendu en spirale (30 min)

Ajoute une option pour rendre les tuiles dans un ordre en spirale depuis le centre de l'image.
Visuellement, ça donne un aperçu du centre de la scène en premier :

```
Tuiles en spirale depuis le centre :
     ┌─────────────────────┐
     │ 16  15  14  13  12 │
     │ 17   4   3   2  11 │
     │ 18   5   0   1  10 │
     │ 19   6   7   8   9 │
     │ 20  21  22  23  24 │
     └─────────────────────┘
```

**Algorithme :** trier les tuiles par distance au centre (`std::sort` + comparateur).

```cpp
auto center_x = width / 2.0, center_y = height / 2.0;
std::sort(tiles.begin(), tiles.end(), [cx, cy](const Tile& a, const Tile& b) {
    auto dist = [cx, cy](const Tile& t) {
        double dx = (t.x0 + t.x1) / 2.0 - cx;
        double dy = (t.y0 + t.y1) / 2.0 - cy;
        return dx*dx + dy*dy;
    };
    return dist(a) < dist(b);
});
```

## Exercice 6 (bonus) — Work stealing (60 min)

Dans le thread pool actuel, si la queue centrale est vide mais qu'un thread a une longue tâche, d'autres threads sont idle.

Le **work stealing** : chaque thread a sa propre queue. Quand sa queue est vide, il "vole" une tâche dans la queue d'un autre thread.

**Structure :**
```cpp
class WorkStealingPool {
    struct Worker {
        std::deque<std::function<void()>> local_queue;
        std::mutex local_mutex;
        std::thread thread;
    };
    std::vector<Worker> workers;
};
```

**Algorithme de vol :**
```
1. Prend la prochaine tâche de ma queue locale (back)
2. Si vide :
    pour chaque autre worker w :
        try_lock(w.local_mutex)
        si succès et w.local_queue non vide :
            vole la première tâche (front)
            unlock
            exécute
        si toutes les queues sont vides : wait(cv)
```

Mesure la différence de performance vs le thread pool simple sur un rendu déséquilibré.
