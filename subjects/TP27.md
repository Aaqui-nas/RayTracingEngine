# TP27 — Allocateurs custom et pools mémoire

**Durée estimée : ~10h**
**C++ introduit :** Placement new, mémoire alignée, `std::pmr::memory_resource`, `std::pmr::vector`
**Feature graphique :** Construction du BVH sans allocations heap — zéro fragmentation sur le hot path

---

## Objectif

Le BVH actuel alloue chaque nœud avec `std::make_shared<BVHNode>` — une allocation `new` par nœud.  
Pour un BVH de 10 000 triangles, ça représente ~10 000 allocations séparées sur le heap, dispersées en mémoire. Les accès non-locaux tuent le cache L1/L2.

Ce TP introduit l'**allocation par pool** : allouer un grand bloc de mémoire une seule fois, puis distribuer des morceaux de ce bloc en O(1) sans aller sur le heap système.

Tu vas apprendre :
- **Placement new** — construire un objet à une adresse arbitraire
- **Allocateur linéaire** — le plus simple et le plus rapide des allocateurs
- **`std::pmr`** — Polymorphic Memory Resource : brancher des allocateurs custom sur les conteneurs STL

---

## 1. Placement new

Le `new` ordinaire fait deux choses : **allouer** de la mémoire, puis **construire** l'objet.  
Le `placement new` fait seulement la deuxième : construire à une adresse déjà fournie.

```cpp
// new ordinaire
BVHNode* node = new BVHNode(left, right, bbox);

// Placement new
void* buffer = malloc(sizeof(BVHNode));
BVHNode* node = new (buffer) BVHNode(left, right, bbox);
// ↑ construit l'objet à l'adresse buffer, sans allouer
```

**Destruction :** avec le placement new, il faut appeler le destructeur manuellement :
```cpp
node->~BVHNode();   // appel explicite du destructeur
free(buffer);       // libère la mémoire séparément
```

**Règle :** `delete node` ne fonctionnerait pas correctement — il faut être symétrique : placement new → destructeur manuel.

## 2. Allocateur linéaire (bump allocator)

Un allocateur linéaire maintient juste un pointeur dans un bloc pré-alloué :

```
[ utilisé | utilisé | utilisé | libre...                ]
                              ^
                              bump_ptr
```

Chaque allocation incrémente `bump_ptr` de la taille demandée. **Libération individuelle impossible** — on libère tout le pool d'un coup.

```
Allouer n octets :
    ptr = bump_ptr
    bump_ptr += align_up(n, alignement)
    return ptr

Libérer tout :
    bump_ptr = debut_du_bloc
```

C'est le plus rapide possible : une addition et un return.

---

## 3. std::pmr — Polymorphic Memory Resource

C++17 ajoute `<memory_resource>` qui permet de brancher n'importe quel allocateur sur les conteneurs STL via une interface polymorphique.

```cpp
#include <memory_resource>

// Ressource mémoire custom
class MonPool : public std::pmr::memory_resource {
    void* do_allocate(std::size_t bytes, std::size_t alignment) override;
    void  do_deallocate(void* p, std::size_t bytes, std::size_t alignment) override;
    bool  do_is_equal(const memory_resource& other) const noexcept override;
};

// Utilisation avec un vector PMR
MonPool pool;
std::pmr::vector<BVHNode> nodes(&pool);   // utilise pool pour les allocations
```

Les types `std::pmr::vector`, `std::pmr::string`, `std::pmr::unordered_map` sont identiques à leurs versions non-pmr, mais prennent un `memory_resource*` en paramètre.

## Exercice 1 — Allocateur linéaire de base (60 min)

Crée `src/core/linear_allocator.h` :

```cpp
class LinearAllocator {
    std::byte* buffer;
    std::byte* bump_ptr;
    std::size_t capacity;

public:
    explicit LinearAllocator(std::size_t capacity_bytes);
    ~LinearAllocator();

    // Alloue `size` octets alignés sur `alignment` octets
    void* allocate(std::size_t size, std::size_t alignment = alignof(std::max_align_t));

    // Remet le pointeur au début (libère tout)
    void reset();

    // Mémoire utilisée / totale
    std::size_t used() const;
    std::size_t total() const;
};
```

**Algorithme d'allocation :**

```
addr_actuelle = (intptr_t)bump_ptr
addr_alignee  = (addr_actuelle + alignment - 1) & ~(alignment - 1)
                  ↑ arrondit à la prochaine valeur multiple de alignment
si addr_alignee + size > (intptr_t)(buffer + capacity) :
    throw std::bad_alloc{}

ptr      = (void*)addr_alignee
bump_ptr = (std::byte*)(addr_alignee + size)
return ptr
```

**Test :**
```cpp
LinearAllocator alloc(1024);
void* p1 = alloc.allocate(64, 16);   // adresse alignée sur 16
void* p2 = alloc.allocate(32, 32);   // adresse alignée sur 32
assert((uintptr_t)p1 % 16 == 0);
assert((uintptr_t)p2 % 32 == 0);
alloc.reset();   // p1 et p2 sont invalides, mais pas de fuite
```

## Exercice 2 — Pool d'objets typé (45 min)

Un pool typé simplifie l'API : les objets sont de taille fixe, l'alignement est automatiquement `alignof(T)`.

```cpp
template<typename T>
class ObjectPool {
    LinearAllocator alloc;

public:
    explicit ObjectPool(std::size_t max_objects)
        : alloc(max_objects * sizeof(T)) {}

    template<typename... Args>
    T* construct(Args&&... args) {
        void* mem = alloc.allocate(sizeof(T), alignof(T));
        return new (mem) T(std::forward<Args>(args)...);
        // ↑ placement new : construit T à l'adresse mem
    }

    // Détruire tous les objets et remettre le compteur à zéro
    // (appelle les destructeurs sur tous les objets construits)
    void destroy_all();
};
```

**Difficulté :** `destroy_all` doit appeler `~T()` sur tous les objets construits. Comment les retrouver ?

**Indice :** maintiens une `std::vector<T*>` de tous les pointeurs retournés par `construct`. Dans `destroy_all`, itère dessus en appelant `ptr->~T()`, puis appelle `alloc.reset()`.

## Exercice 3 — BVHPool : BVH sans allocations heap (90 min)

Remplace les `std::make_shared<BVHNode>` par des constructions dans un pool.

**Classe BVHPool :**
```cpp
class BVHPool {
    ObjectPool<BVHNode> pool;

public:
    explicit BVHPool(int max_objects)
        : pool(2 * max_objects) {}  // un BVH a au plus 2N-1 nœuds

    BVHNode* build(std::vector<std::shared_ptr<Shape>>& shapes, int start, int end);
};
```

**Modification de BVHNode :**

Au lieu de retourner `std::shared_ptr<BVHNode>`, les fils sont des `BVHNode*` bruts gérés par le pool :

```cpp
class BVHNode : public Shape {
    BVHNode* left_node  = nullptr;   // fils gauche (si Shape interne)
    BVHNode* right_node = nullptr;
    std::shared_ptr<Shape> left_leaf  = nullptr;  // feuille
    std::shared_ptr<Shape> right_leaf = nullptr;
    AABB bbox;
    // ...
};
```

Ou plus proprement, garde la même interface avec `shared_ptr` pour les feuilles, mais que le **nœud lui-même** est alloué dans le pool (pas de `shared_ptr` vers les nœuds internes).

**Benchmark :**

```cpp
static void BM_BVH_Build_Heap(benchmark::State& state) { /* version actuelle avec make_shared */ }
static void BM_BVH_Build_Pool(benchmark::State& state) { /* version avec pool */ }
```

Mesure la différence sur 1000, 10 000, 100 000 triangles.

## Exercice 4 — std::pmr::memory_resource (60 min)

Pour intégrer ton pool avec la STL (et pouvoir l'utiliser avec `std::pmr::vector`), implémente l'interface `std::pmr::memory_resource` :

```cpp
class LinearResource : public std::pmr::memory_resource {
    LinearAllocator alloc;

protected:
    void* do_allocate(std::size_t bytes, std::size_t alignment) override {
        return alloc.allocate(bytes, alignment);
    }

    void do_deallocate(void*, std::size_t, std::size_t) override {
        // linéaire : pas de libération individuelle
    }

    bool do_is_equal(const memory_resource& other) const noexcept override {
        return this == &other;
    }

public:
    explicit LinearResource(std::size_t capacity) : alloc(capacity) {}
    void reset() { alloc.reset(); }
};
```

**Usage avec `std::pmr::vector` :**

```cpp
LinearResource resource(1024 * 1024);   // 1 Mo
std::pmr::vector<AABB> aabbs(&resource);
aabbs.reserve(10000);   // alloue dans le pool, pas sur le heap

for (auto& shape : shapes) {
    auto bb = shape->bounding_box();
    if (bb) aabbs.push_back(*bb);
}
// resource.reset() → tout est libéré d'un coup
```

Mesure la différence vs `std::vector<AABB>` standard pour 100 000 éléments.

## Exercice 5 — monotonic_buffer_resource (30 min)

La STL fournit déjà un allocateur linéaire PMR : `std::pmr::monotonic_buffer_resource`.

```cpp
// Buffer statique sur la pile
std::byte stack_buf[4096];
std::pmr::monotonic_buffer_resource resource(stack_buf, sizeof(stack_buf));

std::pmr::vector<int> v(&resource);
v.push_back(1);   // alloue dans stack_buf, pas sur le heap !
```

## Exercice 6 — Stack allocator (optionnel, 30 min)

Un stack allocator est comme un allocateur linéaire mais permet de **libérer le dernier bloc** (LIFO) :

```cpp
class StackAllocator {
    LinearAllocator base;
    struct Marker { std::size_t offset; };

public:
    Marker mark() const;             // sauvegarde la position actuelle
    void free_to(Marker m);          // libère tout depuis le marker
    void* allocate(std::size_t size, std::size_t alignment);
};
```

Usage typique : allouer des données temporaires pour un frame de rendu, libérer tout à la fin du frame.
