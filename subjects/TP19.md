# TP19 — Éclairage direct : Shadow rays et NEE

## Objectif

Le path tracer actuel place les sources lumineuses comme de simples sphères émissives. Un rayon ne
les touche que par **hasard** — c'est très bruyant pour les petites lumières. Ce TP introduit
l'**éclairage direct explicite** (Next Event Estimation) : à chaque rebond, on lance un rayon
directement vers les lumières au lieu d'attendre la chance.

Ce TP introduit :
- Les **lambdas complexes** : factories de lambdas, closures capturant d'autres closures
- Le **shadow ray** : tester l'occlusion entre surface et lumière
- Le **NEE (Next Event Estimation)** : éclairage direct à chaque intersection
- La **lumière d'aire sphérique** : échantillonnage d'un cône solide
- Le **MIS (Multiple Importance Sampling)** : combiner NEE et BRDF sampling

---

## 1. Le problème — bruit autour des petites lumières

Avec le path tracing pur, un rayon ne touche une source lumineuse que si sa direction aléatoire
pointe exactement dessus. Pour une lampe de rayon 0.2 à 5 m de distance :

```
angle solide ≈ π × 0.2² / 5² ≈ 0.005 sr
sphère entière = 4π ≈ 12.6 sr
probabilité de toucher la lampe = 0.04 %
```

Il faut **2 500 rayons** en moyenne pour en toucher un. Résultat : bruit extrême avec peu de
samples, et images encore granuleuses avec 256 samples.

La solution : **forcer** un rayon vers chaque lumière à chaque rebond et ajouter sa contribution
directement, pondérée par la probabilité de l'avoir choisi.

---

## 2. Shadow ray — tester l'occlusion

Un shadow ray vérifie si le point de surface `P` est **directement visible** depuis un point `L`
sur la lumière. L'algorithme :

```
dir = normalize(L - P)
dist = |L - P|
si scene.hit(Ray(P + ε·normal, dir), ε, dist - ε) a un résultat
    → P est à l'ombre (blocked)
sinon
    → P est éclairé directement
```

Le `ε` (typiquement 1e-4) évite l'**auto-intersection** : sans lui, le rayon toucherait la surface
d'où il part.

**Interface à ajouter dans `Scene`** :

```cpp
// Retourne true si un objet occupe le segment [from, to)
bool occluded(const Vec3d& from, const Vec3d& to) const;
```

## 3. NEE — Next Event Estimation

### La formule

À chaque intersection diffuse, on calcule l'éclairage direct via un sample de lumière :

```
L_direct = f_r(ω_o, ω_i) × L_e(ω_i) × cos(θ_i) / pdf(ω_i)
```

avec :
- `f_r` : la BRDF — pour du diffus Lambertien, `f_r = albedo / π`
- `L_e(ω_i)` : la radiance émise par la lumière dans la direction ω_i
- `cos(θ_i)` : cosinus de l'angle d'incidence (`dot(direction, normal)`)
- `pdf(ω_i)` : densité de probabilité d'avoir choisi cette direction

### Le problème du double comptage

Si on ajoute l'éclairage direct **et** qu'on laisse aussi les rayons rebondissants toucher les
lumières par hasard, on compte l'énergie deux fois.

Solutions :
- **Option A** (simple, ce TP) : désactiver l'émission des lumières comme contribution indirecte.
  Les lumières sont visibles directement depuis la caméra (primaire), mais ne contribuent pas via
  les rebonds — légère approximation acceptable pour commencer.
- **Option B** (section 8, extension) : MIS — pondérer les deux stratégies correctement.

### Structure `LightSample`

```cpp
struct LightSample {
    Vec3d  direction;  // direction normalisée depuis le point de surface vers la lumière
    double distance;   // distance jusqu'au point d'impact sur la lumière
    Vec3d  radiance;   // radiance émise (valeurs HDR, non gamma-corrigées)
    double pdf;        // densité de probabilité en angle solide [sr⁻¹]
};
```

## 4. Lumière d'aire sphérique — géométrie

### Angle solide d'une sphère

Vue depuis un point `p`, une sphère de centre `c` et rayon `r` (avec `d = |c-p| > r`)
subtend un **cône** de demi-angle `θ_max` :

```
sin(θ_max) = r / d
cos(θ_max) = sqrt(1 - (r/d)²)
```

L'angle solide de ce cône vaut : `Ω = 2π × (1 - cos(θ_max))`

### Échantillonnage uniforme dans le cône

Pour échantillonner uniformément une direction dans ce cône :

```
cos_θ = 1 + ξ₁ × (cos_θ_max - 1)          // uniforme dans [cos_θ_max, 1]
sin_θ = sqrt(1 - cos_θ²)
φ     = 2π × ξ₂

direction_locale = (sin_θ × cos(φ),  sin_θ × sin(φ),  cos_θ)
```

La direction locale a pour axe z l'axe vers le centre de la sphère. Il faut la
**transformer en espace monde** via un repère orthonormé local.

### Construction du repère local

Soit `z = normalize(c - p)` (direction vers le centre de la sphère).

Pour trouver un vecteur `x` perpendiculaire à `z` :
- Choisir un vecteur "up" arbitraire `a` — `(0,1,0)` ou `(1,0,0)` selon si `z` est proche de y
- `x = normalize(cross(z, a))`
- `y = cross(z, x)`

Alors `direction_monde = sin_θ cos_φ × x + sin_θ sin_φ × y + cos_θ × z`

### PDF

L'échantillonnage est uniforme sur le cône ⟹ pdf constante :

```
pdf = 1 / Ω = 1 / (2π × (1 - cos_θ_max))
```

### Distance à la surface

Une fois la direction `dir` échantillonnée, calculer où elle touche la sphère via
l'équation quadratique standard (comme dans `Sphere::hit`). Stocker `t` comme `distance`.

## 5. C++ — lambdas complexes comme objets Light

### Le patron déjà établi

Dans le renderer, les matériaux sont des lambdas :

```cpp
using Material = std::function<optional<Scatter>(const Ray&, const HitRecord&)>;

auto metal = [](Vec3d color, double fuzz) -> Material {
    return [color, fuzz](const Ray& r, const HitRecord& rec) -> optional<Scatter> { ... };
};
```

`metal` est une **factory de lambda** : une fonction qui retourne un lambda capturant son état.

### Light comme paire de lambdas

Pour les lumières, on a besoin de **deux fonctions** liées partageant le même état :

```cpp
struct Light {
    std::function<std::optional<LightSample>(const Vec3d& from, double xi1, double xi2)> sample;
    std::function<double(const Vec3d& from, const Vec3d& direction)>                     pdf;
};
```

`make_sphere_light` est une factory qui retourne un `Light` dont les deux membres
`sample` et `pdf` sont des **lambdas capturant `center`, `radius` et `emission` par valeur** :

```cpp
Light make_sphere_light(Vec3d center, double radius, Vec3d emission) {
    return Light {
        .sample = [center, radius, emission](const Vec3d& from, double xi1, double xi2)
                     -> std::optional<LightSample> { /* ... */ },
        .pdf    = [center, radius](const Vec3d& from, const Vec3d& direction)
                     -> double { /* ... */ }
    };
}
```

Les deux lambdas partagent `center` et `radius` via leurs captures indépendantes — pas de
pointeur partagé nécessaire.

**Intérêt** : la `LightList` peut stocker des `Light` hétérogènes (sphères, disques,
triangles lumineux) derrière la même interface polymorphe, sans héritage.

## 6. LightList — plusieurs sources de lumière

```cpp
struct LightList {
    std::vector<Light> lights;

    void add(Light l)  { lights.push_back(std::move(l)); }
    bool empty() const { return lights.empty(); }
    int  size()  const { return (int)lights.size(); }

    // Sélectionne une lumière au hasard (via xi0), puis l'échantillonne
    std::optional<LightSample> sample(const Vec3d& from,
                                      double xi0, double xi1, double xi2) const;

    // PDF combinée = moyenne des PDFs individuelles
    double pdf(const Vec3d& from, const Vec3d& direction) const;
};
```

### Sélection uniforme

En choisissant chaque lumière avec probabilité `1/N` (où `N = lights.size()`) :

```
idx   = (int)(xi0 * N)    // ∈ [0, N-1]
ls    = lights[idx].sample(from, xi1, xi2)
ls.pdf /= N               // correction pour la sélection
```

La PDF d'avoir choisi cette direction est `light_pdf / N`, car il y a `N` choix possibles.

### PDF combinée

```
pdf_combinée(from, dir) = (1/N) × Σᵢ lights[i].pdf(from, dir)
```


### Modification de `ray_color()`

```cpp
static Vec3d ray_color(const Ray& ray, const Scene& scene, int depth) {
    auto sky = [&]() -> Vec3d { /* ... (inchangé) */ };

    if (depth == 0) return sky();

    if (auto rec = scene.hit(ray, 0.001, 1000)) {
        // Option A : ne pas propager l'émission des rebonds non-primaires
        Vec3d emitted = /* seulement si rayon primaire */ Vec3d(0,0,0);

        // Éclairage direct (NEE)
        Vec3d direct(0, 0, 0);
        if (!scene.lights.empty()) {
            // Générer trois nombres aléatoires
            auto ls = scene.lights.sample(rec->point, rng(), rng(), rng());
            if (ls && ls->pdf > 0) {
                Vec3d shadow_origin = rec->point + 1e-4 * rec->normal;
                Vec3d light_point   = rec->point + ls->direction * ls->distance;
                if (!scene.occluded(shadow_origin, light_point)) {
                    double cos_theta = std::max(0.0, dot(ls->direction, rec->normal));
                    // Pour du Lambertien : f_r = attenuation / (π × 255)
                    // direct = f_r × radiance × cos_theta / pdf × 255
                    // ...
                }
            }
        }

        // Éclairage indirect (rebond)
        if (auto scatter = rec->material(ray, *rec))
            return direct + emitted
                   + mul(scatter->attenuation / 255.0,
                         ray_color(scatter->scattered, scene, depth - 1));
        return direct + emitted;
    }
    return sky();
}
```

> **Attention** : le facteur 255 est présent parce que les couleurs de matériaux sont en [0,255]
> dans ce renderer, et la radiance de l'env map aussi. La cohérence des unités est importante.

## 8. MIS — Multiple Importance Sampling (extension)

### Le problème des deux stratégies

Avec NEE pure (Option A), les lumières ne contribuent plus comme radiance indirecte. Pour voir
les reflets d'une lampe dans un miroir, ou les caustiques, il faudrait Option B : conserver les
deux stratégies sans les compter deux fois.

Le **MIS (Veach 1995)** pondère les contributions pour qu'elles somment correctement :

```
L = w_nee × L_nee + w_brdf × L_brdf
```

### Heuristique de balance

La pondération balance (n=1) pour deux stratégies :

```
w_a(ω) = pdf_a(ω) / (pdf_a(ω) + pdf_b(ω))
```

Elle satisfait `w_a(ω) + w_b(ω) = 1` pour tout ω.

### Heuristique de puissance (n=2, recommandée)

```
w_a(ω) = pdf_a²(ω) / (pdf_a²(ω) + pdf_b²(ω))
```

Généralement meilleure variance que la balance heuristic.

### Interface

```cpp
// Heuristique de puissance n=2
double mis_weight(double pdf_a, double pdf_b);
```

Propriétés vérifiables :
- `mis_weight(a, b) + mis_weight(b, a) == 1.0` pour tout a, b > 0
- `mis_weight(a, 0) == 1.0`
- `mis_weight(0, b) == 0.0`

### Intégration dans `ray_color()`

NEE avec MIS :
```
w_nee  = mis_weight(ls.pdf, brdf_pdf)
L_nee  = w_nee × (f_r × L_e × cos_θ) / ls.pdf
```

Rebond BRDF avec MIS :
```
pdf_brdf = cos_theta / π   (pour Lambertien)
w_brdf   = mis_weight(pdf_brdf, scene.lights.pdf(point, scattered_dir))
L_brdf   = w_brdf × ray_color(scattered_ray, ...) × albedo
```
