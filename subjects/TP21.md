## Pourquoi les concepts ?

En C++ avant C++20, écrire du code générique reposait sur les templates "nus".  
Si tu passais un mauvais type, tu obtenais une erreur de compilateur de 80 lignes pointant sur les entrailles de la STL.

```cpp
// Template nu — C++17
template<typename T>
void render_shape(const T& shape, const Ray& ray) {
    shape.hit(ray, 0.001, 1000.0);  // compile seulement si T a un hit()
}
```

Si tu appelles `render_shape(42, ray)`, le compilateur te répond :
```
error: 'const int' has no member named 'hit'
  --> ...dans ...dans ...dans ...dans ...
     template instantiation
```

**Le concept** te permet d'exprimer l'exigence explicitement :

```cpp
// C++20 avec concept
template<Hittable T>
void render_shape(const T& shape, const Ray& ray) {
    shape.hit(ray, 0.001, 1000.0);
}
```

Erreur si le type ne convient pas :
```
error: 'int' does not satisfy 'Hittable'
note: requires expression failed
```

Une ligne. Lisible. C'est le but des concepts.

## Syntaxe de base

### Définir un concept

```cpp
template<typename T>
concept NomDuConcept = /* expression booléenne à la compilation */ ;
```

L'expression la plus courante est un **`requires` expression** :

```cpp
template<typename T>
concept Printable = requires(T t) {
    { std::cout << t };          // t doit être "printable"
};

template<typename T>
concept Sized = requires(T t) {
    { t.size() } -> std::convertible_to<std::size_t>;  // retourne quelque chose convertible en size_t
};
```

### Utiliser un concept

Quatre syntaxes équivalentes :

```cpp
// Syntaxe 1 : concept comme "type"
template<Printable T>
void print(T value);

// Syntaxe 2 : requires en fin de signature
template<typename T>
void print(T value) requires Printable<T>;

// Syntaxe 3 : auto avec concept
void print(Printable auto value);

// Syntaxe 4 : requires inline
template<typename T> requires Printable<T>
void print(T value);
```

Dans ce TP, tu utiliseras surtout 1 et 3.

## Exercice 1 — Concept `Hittable` (30 min)

Crée le fichier `src/geometry/concepts.h`.

Un objet est *hittable* s'il possède une méthode `hit(ray, tmin, tmax)` qui retourne `std::optional<HitRecord>`.

**À faire :** Définis le concept `Hittable` dans `namespace rt` :

```cpp
template<typename T>
concept Hittable = requires(const T& t, const Ray& r, double tmin, double tmax) {
    // exprime que t.hit(r, tmin, tmax) doit retourner std::optional<HitRecord>
    { ... } -> std::same_as<std::optional<HitRecord>>;
};
```

**Vérification :** Dans le fichier, juste en dessous, ajoute :
```cpp
static_assert(Hittable<Sphere>);
static_assert(Hittable<Plane>);
static_assert(!Hittable<int>);
```

Si ça compile → le concept est correct.

## Exercice 2 — Concept `Bounded` (20 min)

Un objet est *bounded* s'il est `Hittable` **et** possède `bounding_box()` retournant `std::optional<AABB>`.

**À faire :** Définis `Bounded` en termes de `Hittable` :

```cpp
template<typename T>
concept Bounded = Hittable<T> && requires(const T& t) {
    { t.bounding_box() } -> std::same_as<std::optional<AABB>>;
};
```

C'est la **subsomption** : `Bounded` est plus fort que `Hittable`.  
Si les deux se superposent dans une surcharge, le compilateur préfère la plus spécialisée.

**Vérification :**
```cpp
static_assert(Bounded<Sphere>);
static_assert(!Bounded<Plane>);   // Plane n'a pas de bounding_box
```

## Exercice 3 — Fonction contrainte : `closest_hit` (30 min)

Actuellement, `Scene::hit()` itère sur `std::vector<std::shared_ptr<Shape>>`.  
Écris une fonction générique qui peut travailler sur **n'importe quel range de Hittable**.

**À faire :** Dans `src/geometry/concepts.h`, ajoute :

```cpp
// Concept pour un range de Hittable (pointeurs ou valeurs)
template<typename R>
concept HittableRange = std::ranges::range<R> && requires(R r) {
    requires Hittable<std::remove_cvref_t<decltype(*std::ranges::begin(r))>>;
};
```

Puis la fonction :
```cpp
template<HittableRange R>
std::optional<HitRecord> closest_hit(const R& objects, const Ray& ray,
                                      double tmin, double tmax) {
    // parcourt objects, retourne le hit avec le t le plus petit
}
```

**Indice :** le corps est identique à `Scene::hit()` sans le BVH — copie et adapte.

**Test :** Crée un `std::vector<Sphere>` (pas de `shared_ptr` !), appelle `closest_hit` dessus.  
Vérifie que `closest_hit(std::vector<int>{}, ray, 0, 1)` ne compile **pas**.

## Exercice 4 — `auto` paramètre et `std::ranges` (40 min)

C++20 autorise `auto` comme paramètre de fonction, avec un concept optionnel.  
C'est du sucre syntaxique pour un template contraint.

```cpp
void f(Hittable auto& shape) { /* template implicite */ }
// équivalent à :
template<Hittable T>
void f(T& shape) {}
```

**À faire :**

**4a.** Écris une fonction `emissive_shapes` qui filtre les `shared_ptr<Shape>` émissifs :

```cpp
auto emissive_shapes(const std::vector<std::shared_ptr<Shape>>& objects) {
    return objects | std::views::filter([](const auto& obj) {
        return obj->emission.length() > 0.0;
    });
}
```

Observe : `std::views::filter` retourne une **view** paresseuse — aucune copie n'est faite.

**4b.** Écris une fonction `bounded_shapes` qui ne garde que ceux qui ont une AABB :

```cpp
auto bounded_shapes(const std::vector<std::shared_ptr<Shape>>& objects) {
    return objects | std::views::filter([](const auto& obj) {
        return /* ... */;
    });
}
```

**4c.** Utilise `std::ranges::sort` pour trier les `shared_ptr<Shape>` par centre de leur AABB :

```cpp
void sort_by_axis(std::vector<std::shared_ptr<Shape>>& objects, int axis) {
    std::ranges::sort(objects, [axis](const auto& a, const auto& b) {
        auto ba = a->bounding_box();
        auto bb = b->bounding_box();
        if (!ba || !bb) return false;
        return ba->center()[axis] < bb->center()[axis];
    });
}
```

Compare avec la version actuelle dans `BVHNode` — c'est exactement ce qu'elle fait, mais en une ligne.

## Exercice 6 — Concept composé : `Renderable` (30 min)

Un objet *renderable* est à la fois :
- `Hittable` (peut être intersecté)
- Il expose une `Material` (possède un champ ou accesseur `material`)

**À faire :** Définis `Renderable` :

```cpp
template<typename T>
concept Renderable = Hittable<T> && requires(const T& t) {
    { t.material } -> std::convertible_to<Material>;
};
```

## Exercice 7 (bonus) — Refactorer `build_bvh` avec `std::ranges` (40 min)

Dans `scene.h`, la méthode `build_bvh()` sépare les objets bornés des non-bornés avec une boucle manuelle.

**À faire :** Réécris-la avec `std::ranges::partition_copy` ou `std::ranges::copy_if` :

```cpp
void build_bvh() {
    std::vector<std::shared_ptr<Shape>> bounded, unbounded;
    
    std::ranges::partition_copy(objects,
        std::back_inserter(bounded),
        std::back_inserter(unbounded),
        [](const auto& obj) { return obj->bounding_box().has_value(); });
    
    objects = unbounded;
    if (!bounded.empty())
        bvh_root = std::make_shared<BVHNode>(bounded, 0, (int)bounded.size());
}
```

Assure-toi que le rendu produit le même résultat qu'avant.

## Ce que tu as appris

| Concept C++ | Ce que ça fait |
|---|---|
| `concept Name = requires(T t) { ... }` | Définit une interface compile-time |
| `{ expr } -> std::same_as<T>` | Contraint le type de retour |
| `template<Concept T>` | Applique le concept à un paramètre template |
| `Concept auto` | Sucre syntaxique pour template contraint |
| `static_assert(Concept<T>)` | Vérifie à la compilation avec message |
| `std::ranges::sort` | Sort un range avec comparateur |
| `std::views::filter` | View paresseuse filtrée |
| Subsomption | Le concept le plus spécialisé est préféré dans les surcharges |
